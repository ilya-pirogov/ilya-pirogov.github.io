Critical Findings
Critical Stored XSS via Leaderboard flag Field

Description: The leaderboard API (/api/leaderboard) accepts arbitrary strings in the flag field with no server-side sanitization. The client-side flagHtml() function in leaderboard.js falls through to returning the raw value when it doesn't match known patterns (2-letter country code, emoji, or "MARS"). This raw value is then injected into the DOM via innerHTML in renderGlobalRecords().

Attack vector: An attacker can POST a crafted payload to the API. Every user who loads the leaderboard will execute the attacker's script.

// Proof of concept — accepted by the API with HTTP 200
POST /api/leaderboard
{
  "name": "XSS-001",
  "score": 50,
  "flag": "<script>alert(document.cookie)</script>",
  "distance": 100
}
// Response: {"ok":true,"rank":619,"total":622}

Vulnerable code path:

// leaderboard.js — flagHtml() returns unsanitized input
export function flagHtml(flag) {
  if (!flag) return '';
  if (flag === MARS_FLAG) return '<img ...>';
  let code = flag;
  if (flag.length > 2) code = emojiToCode(flag);
  if (/^[A-Za-z]{2}$/.test(code)) return `<span class="fi ..."></span>`;
  return flag;  // ← RAW HTML returned for non-matching input
}

// renderGlobalRecords() inserts via innerHTML
el.innerHTML = records.map((r, i) => `
  ...${flagHtml(r.flag)}...  // ← XSS injection point
`).join('');

Impact: Session hijacking, credential theft, defacement of the leaderboard, phishing overlays, or redirecting users to malicious sites. Affects all users who view the leaderboard.

Fix: Validate the flag field server-side — reject anything that isn't a 2-letter ISO country code or "MARS". Additionally, escape HTML in flagHtml() before returning, or use textContent instead of innerHTML.
Critical Leaderboard Score Injection — No Server-Side Validation

Description: The leaderboard API accepts arbitrary score and distance values with no server-side plausibility checks, rate limiting, or authentication. Anyone can submit fabricated scores directly via HTTP POST.

// Submitted a score of 999,999,999 — 90x higher than the #1 player
POST /api/leaderboard
{ "name": "TST-999", "score": 999999999, "flag": "US", "distance": 999999 }
// Response: {"ok":true,"rank":1,"total":620}

// Also accepted: distance of 99,999,999,999
POST /api/leaderboard
{ "name": "BUG-002", "score": 100, "flag": "US", "distance": 99999999999 }
// Response: {"ok":true,"rank":588,"total":621}

What IS validated: Callsign format (AAA-000) and score type (must be a positive number). What is NOT validated: Score magnitude, distance plausibility, rate limiting, duplicate submissions, authentication, or any proof the score was earned in-game.

Impact: Complete leaderboard integrity destruction. Any user can claim rank #1 at will, undermining the competitive aspect of the game.

Fix: Implement server-side score validation (max plausible score caps based on game mechanics), rate limiting per IP/callsign, HMAC-signed score submissions from the client, and/or anti-tamper checks tied to game session data.

High Full Debug/God Mode Accessible in Production

Description: The complete debug system (file: debug.js, ~700 lines) is loaded in the production build. Pressing Alt+L twice activates a full debug panel with:

KeyCheat
GGod Mode — invincible ship
FInfinite fuel and zero heat
M+50,000 credits instantly
DSet debt to 100 CR (skip 20,000 CR grind)
BSpawn landing pad at ship location
] / [Teleport 2000px forward/backward
1-4Switch levels instantly
TSlow-mo (0.2x time scale)
Shift+1-5Trigger storms at will

The debug panel also exposes a Performance Profiler, Terrain Designer, AI Pilot, Flight Recorder, and Asset Catalog — all developer-only tools.

Impact: Any player can activate god mode and cheat to earn inflated scores, then submit them to the global leaderboard. Combined with the lack of server-side score validation, this completely undermines gameplay integrity.

Fix: Strip debug.js and all debug UI elements from production builds. Use a build-time flag or environment variable to exclude debug code.
High Premium/Ad-Free Status Bypass via localStorage

Description: The premium status check relies entirely on a localStorage flag. The premium.js module reads localStorage.getItem('uber_mars_is_premium') and trusts it unconditionally on web platforms (where no native billing is available).

// premium.js — _grantPremium()
function _grantPremium() {
  _isPremium = true;
  localStorage.setItem('uber_mars_is_premium', '1');  // ← trivially spoofable
}

// Bypass: run in browser console
localStorage.setItem('uber_mars_is_premium', '1');
// Reload page → Premium.isPremium === true

On the web platform, the isNative() check returns false, meaning billing is never initialized and the only guard is the localStorage value. Setting it grants instant premium status, removing all ads and enabling offline mode.

Impact: Complete bypass of the $3.99 premium purchase. All ad-gated rewards (continue after death, 500 CR bonus, fine reduction) are granted instantly without viewing ads.

Fix: Server-side purchase verification. On web, premium should be tied to an authenticated account with receipt validation, not a client-side flag.
High Complete Client-Side Game State Manipulation

Description: All game state is stored in mutable JavaScript objects exported from state.js and directly accessible via the browser console. There is no obfuscation, encryption, or integrity checking.

// All state objects are globally accessible via ES module imports
// From browser console (via debug or module scope):
player.credits = 999999;       // infinite money
player.debtRemaining = 0;      // skip entire debt mechanic
player.buyoutDone = true;      // instant endgame
ship.fuel = Infinity;          // infinite fuel
ship.gearHealth = Infinity;    // indestructible
ship.maxFuel = 99999;          // massive fuel tank

Impact: Players can trivially modify any game variable — credits, debt, fuel, health, cargo, ship upgrades, story progression — to skip all gameplay and immediately reach endgame content.

Fix: For a client-side game this is partially inherent, but critical values (especially those submitted to the leaderboard) should have server-side integrity checks. Consider obfuscating/minifying production code and adding tamper detection for leaderboard-eligible runs.
Medium Severity Findings
Medium Exposed AdMob Production Ad Unit IDs

Description: Production AdMob ad unit IDs are hardcoded in plain text in ads.js:

const ADMOB_REWARDED_ID       = 'ca-app-pub-2570100164384367/4634303281';
const ADMOB_INTERSTITIAL_ID   = 'ca-app-pub-2570100164384367/6726114156';
const ADMOB_REWARDED_INTER_ID = 'ca-app-pub-2570100164384367/2719061748';

Impact: Competitors or malicious actors could use these IDs to generate fraudulent ad impressions, potentially getting the developer's AdMob account flagged or banned by Google. The publisher ID (pub-2570100164384367) also links to the developer's Google AdSense account.

Fix: Serve ad unit IDs from a server-side configuration endpoint rather than hardcoding them in client-side JavaScript. At minimum, obfuscate them in the build.
Medium Ad Reward Bypass — Placeholder Fallback Grants Free Rewards

Description: When AdMob fails to load (which is always on web), the ads.js system falls back to a 3-second countdown overlay (_showPlaceholderAd) and then calls cb.adRewarded() — granting the reward without any real ad being watched.

// ads.js — Android fallback: no real ad needed
} else if (_platform === 'android' && !_admobAvailable) {
  _showPlaceholderAd({
    adStarted: () => cb.adStarted?.(),
    adCompleted: () => cb.adRewarded?.(),  // ← reward granted after 3s timer
  });
}

On web, rewarded ads call cb.adError('no-ads-on-web'), but the calling code in the game UI may handle this by granting rewards anyway or showing the placeholder. The 3-second placeholder timer can also be bypassed by manipulating the DOM.

Impact: Ad-gated features (continue after death, 500 CR bonus, fine reduction) can be accessed without watching real ads, reducing ad revenue.

Fix: On web, disable ad-reward buttons entirely. On Android, verify ad completion through AdMob's server-side verification (SSV) before granting rewards.
Medium No Content Security Policy (CSP)

Description: The site has no Content Security Policy header or meta tag. This means there are no browser-enforced restrictions on script sources, inline scripts, or resource loading origins.

Impact: If an attacker achieves script injection (e.g., via the stored XSS in the flag field), there are no CSP restrictions to limit the damage. The attacker can load external scripts, exfiltrate data to any domain, and execute arbitrary inline JavaScript.

Fix: Implement a strict CSP header: Content-Security-Policy: default-src 'self'; script-src 'self' cdn.jsdelivr.net; style-src 'self' 'unsafe-inline' cdn.jsdelivr.net; img-src 'self' data:;
Low Severity Findings
Low Unminified Source Code Exposes Full Application Logic

Description: All 80+ JavaScript files are served unminified with full variable names, comments, and clear structure. Files include: state.js, config.js, debug.js, leaderboard.js, premium.js, ads.js, ship.js, game-logic.js, and many more.

Impact: Dramatically lowers the barrier for discovering exploits. An attacker can read the full game logic, identify all state variables, understand the leaderboard API contract, and find every input validation gap without any reverse-engineering effort.

Fix: Use a bundler (Vite, Webpack, Rollup) to bundle and minify production code. This won't prevent determined attackers but raises the effort required.
Low OG/Twitter Meta Tags Reference Old Domain

Description: Open Graph and Twitter Card meta tags still reference ubermars.vercel.app instead of the production domain ubermars.app:

<meta property="og:image" content="https://ubermars.vercel.app/assets/images/poster.png">
<meta property="og:url" content="https://ubermars.vercel.app/">
<meta name="twitter:image" content="https://ubermars.vercel.app/assets/images/poster.png">

Impact: Minor — social media previews may break if the Vercel subdomain is reconfigured. Also leaks the deployment platform (Vercel) and original project name. The leaderboard API also falls back to this domain.

Fix: Update all meta tags and the API_BASE constant to use the production domain ubermars.app.
Summary of Recommendations

Priority 1 — Immediate: Fix the stored XSS in the flag field. Validate all leaderboard inputs server-side (callsign format, flag as ISO country code, score within plausible range). Add a Content Security Policy header.

Priority 2 — Short term: Remove debug.js from production builds. Implement rate limiting on the leaderboard API. Move premium validation server-side or add receipt verification.

Priority 3 — Medium term: Bundle and minify JavaScript. Implement signed score submissions (HMAC). Add server-side ad verification (AdMob SSV). Update meta tags to production domain.

Security audit performed on March 19, 2026. Findings are based on black-box testing of the public web application.
